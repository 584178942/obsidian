## 一、项目概述

### 1.1 项目背景与目标

MDM（Mobile Device Management，移动设备管理）企业移动管理系统是由上海思御信息科技有限公司自主研发的企业级设备管控解决方案，适配华为鸿蒙 NEXT 6.0 及以上操作系统（API 14+）。系统通过 MQTT 协议建立设备与服务器之间的长连接通讯通道，实现企业对移动设备的远程集中管控，满足政企客户对设备安全合规管理的需求。

本系统旨在解决企业移动设备管理中的以下核心痛点：

- **设备安全管控**：支持远程锁定、恢复出厂设置、重启、重置密码等操作，防止设备丢失后的数据泄露；
- **应用生命周期管理**：支持企业应用静默安装、强制卸载及黑名单管理，保障设备仅运行授权应用；
- **网络与外设管控**：支持 WiFi、移动数据、蓝牙、GPS、相机、麦克风、USB 等外设的启用/禁用控制；
- **设备状态监控**：实时采集设备电量、网络类型、屏幕状态、在线状态等关键指标，支持后台运维决策。

## 三、核心模块详细说明

### 3.1 MQTT 通信层（mqtt/）

MQTT 通信层是整个系统的通信中枢，负责与 MQTT Broker 建立长连接并维持设备与服务器之间的实时双向通信。

**MqttConfig.ets（配置参数模块）**

该模块定义了 MQTT 连接所需的所有配置参数，包括服务器地址、连接超时、保活间隔、遗嘱消息（Will Message）等配置。设备端以 `mdm_siyu_{deviceId}` 作为客户端标识前缀，通过设置遗嘱消息实现设备异常离线时的自动通知。当设备正常或异常断开连接时，Broker 会自动向 `mdm/report` 主题发布设备离线通知，服务器据此感知设备在线状态变化。

```typescript
export class MqttConfig {
  static readonly SERVER_URI = 'tcp://192.168.1.199:1883'
  static readonly CLIENT_ID_PREFIX = 'mdm_siyu_'
  static readonly CONNECTION_TIMEOUT = 30      // 秒
  static readonly KEEP_ALIVE_INTERVAL = 60    // 秒
  static readonly CLEAN_SESSION = true
  static readonly AUTO_RECONNECT = true
  static readonly USERNAME = 'admin'
  static readonly PASSWORD = 'admin'
  static readonly WILL_TOPIC = 'mdm/report'
  static readonly WILL_QOS = 1
  static readonly WILL_RETAINED = true
  static readonly PUBLISH_QOS = 1
  static readonly SUBSCRIBE_QOS = 1
}
```

**MqttClient.ets（MQTT 客户端封装模块）**

该模块封装了 MQTT 客户端的初始化、连接、订阅、发布、断开连接等核心操作，并对外提供连接状态监听接口。当设备与 Broker 连接成功时，客户端自动订阅设备专属主题 `mdm/{deviceId}` 和配置更新主题 `device/config/{deviceId}`，确保设备仅接收与其相关的信息。当连接断开时，通过连接状态监听器触发重连流程。该模块支持多个连接状态监听器同时注册，以便其他模块（如心跳模块、命令处理模块）及时感知连接状态变化。

**ReconnectManager.ets（智能重连策略模块）**

为避免频繁重连带来的资源消耗，该模块实现了基于时间的智能重连延迟策略：断开连接后5分钟内每5秒重试一次，5至30分钟每30秒重试一次，30至60分钟每60秒重试一次，1至6小时每3分钟重试一次，超过6小时每5分钟重试一次。通过记录断开连接的时间戳，动态计算下一次重连的等待时间，在保证恢复速度的同时降低服务器端的连接压力。

### 3.2 命令处理系统（command/）

命令处理系统负责解析并执行从服务器下发的各类管控指令，是设备端业务逻辑的核心调度中心。

**CommandParser.ets（命令消息解析模块）**

该模块将服务器下发的 JSON 格式命令消息解析为结构化的 CommandMessage 对象，支持 commandId、action、params 和 timestamp 四个标准字段。params 字段为可选的键值对，模块提供了 `getString`、`getBoolean`、`getNumber` 三个安全取值方法，在参数缺失时返回默认值，避免空指针异常。解析失败的 JSON 或缺少必要字段的命令将被丢弃并返回 null。

**CommandResult.ets（命令结果封装模块）**

该模块负责将命令执行结果封装为统一格式的 JSON 响应消息，包含 commandId、action、success、message、data 和 timestamp 字段。提供了 `success` 和 `failure` 两个静态工厂方法，调用方只需传入命令ID、动作类型和结果描述即可快速生成结果对象。toJson 方法将结果序列化为字符串后通过 MQTT 上报给服务器。

**CommandHandler.ets（命令执行分发模块）**

该模块是命令处理系统的核心调度器，根据 action 字段的值的不同，调用对应的 MDM 功能模块执行实际操作。支持的 action 包括：设备控制类（lock、unlock、wipe、reboot、reset_password）、应用管理类（install_app、uninstall_app）、限制策略类（disable_wifi、enable_wifi、disable_mobile_data、enable_mobile_data、set_gps、set_bluetooth、disable_camera、enable_camera）、信息查询类（get_device_info、get_app_list、sync_config）。对于未知命令，返回包含错误信息的结果对象。

### 3.3 MDM 核心功能模块（mdm/）

MDM 功能模块直接调用鸿蒙 MDM Kit（@kit.MDMKit）提供的企业设备管理能力实现对设备的管控操作。所有操作均需设备已激活 EnterpriseAdminExtensionAbility（即设备管理员权限）方可执行。

**DeviceController.ets（设备控制模块）**

该模块封装了设备远程控制的核心操作，包括：

- **lock(mainText, subText)**：锁定设备屏幕，显示指定提示文字，适用于设备丢失后的紧急锁定场景；
- **unlock()**：解除设备屏幕锁定，恢复正常使用；
- **wipe()**：执行恢复出厂设置，清除设备上所有数据，该操作不可逆；
- **rebootDevice()**：远程重启设备，适用于设备固件升级后的重启生效场景；
- **resetPassword(password)**：远程重置设备解锁密码，防止管理员忘记密码导致设备无法使用。

**AppManager.ets（应用管理模块）**

该模块提供企业应用的全生命周期管理能力：

- **install(apkPath)**：通过指定路径安装应用包，适用于企业内测应用的静默分发；
- **uninstall(bundleName)**：通过应用包名卸载指定应用，支持强制卸载违规应用。

**RestrictionManager.ets（限制策略模块）**

该模块通过 `setDisallowedPolicy` 接口设置设备策略，支持以下限制项：wifi（WiFi 网络）、bluetooth（蓝牙）、camera（相机）、microphone（麦克风）、usb（USB 调试/文件传输）、screenshot（截屏）、hdc（HiDDebug能力）等。通过设置禁用策略，可有效防止企业设备在敏感情景下使用指定功能。

**InfoQuerier.ets（信息查询模块）**

该模块提供设备信息与应用列表的查询能力：

- **getDeviceInfo()**：获取设备制造商、型号、设备名称、操作系统版本、SDK API 版本等基本信息；
- **getAppList()**：获取设备上已安装应用的包名和应用名称列表，支持应用白名单审计。

### 3.4 心跳与电源管理（battery/）

**HeartbeatAdjuster.ets（动态心跳调整模块）**

该模块根据设备当前状态动态调整心跳上报间隔，在保障服务器感知设备在线状态的前提下尽可能降低设备电量消耗：

- 充电中且电量大于50%：每1分钟心跳一次（高性能模式）；
- 电量低于20%且未充电：每5分钟心跳一次（省电模式）；
- 屏幕关闭且未充电：每10分钟心跳一次（超省电模式）；
- 正常使用中：每2分钟心跳一次（普通模式）。

心跳消息内容包含设备ID、电量百分比、充电状态、屏幕状态、网络类型和当前时间戳。

**BatteryMonitor.ets（电池监控模块）**

该模块封装了电池信息的读取和变更监听能力，可实时获取当前电池电量和充电状态，并在电池状态变化时触发回调通知，用于联动心跳调整和电源管理策略。

### 3.5 网络状态监控（NetworkMonitor.ets）

该模块封装了鸿蒙网络连接管理接口（@ohos.net.connection），提供网络可用性监听和同步查询能力。当网络从不可用变为可用时触发回调通知，确保 MQTT 重连、命令下发等依赖网络的操作能够及时响应网络恢复事件。

### 3.6 数据库设计

本地存储采用双存储机制：

- **SQLite（relationalStore）**：存储命令历史记录（command_history 表），包括命令ID、动作类型、参数、状态、结果和时间戳，支持命令执行记录的追溯查询；
- **Preferences（键值存储）**：存储 MQTT 配置、设备ID等关键配置信息，支持快速读写。

## 四、MQTT 消息协议

### 4.1 主题定义

| 主题 | 通信方向 | 用途 |
|------|---------|------|
| `mdm/{deviceId}` | 服务器 → 设备 | 下发管控指令 |
| `device/config/{deviceId}` | 服务器 → 设备 | 下发配置更新 |
| `mdm/report` | 设备 → 服务器 | 设备离线通知（遗嘱） |
| `mdm/command/{deviceId}` | 设备 → 服务器 | 上报命令执行结果 |
| `mdm/{deviceId}/heartbeat` | 设备 → 服务器 | 设备心跳数据 |
| `mdm/{deviceId}/alert` | 设备 → 服务器 | 设备告警信息 |

### 4.2 命令消息格式

服务器下发命令采用 JSON 格式，示例：

```json
{
  "commandId": "uuid-123456",
  "action": "lock",
  "params": {
    "mainText": "设备已锁定",
    "subText": "请联系管理员"
  },
  "timestamp": 1704067200000
}
```

### 4.3 支持的命令列表

| action | 功能说明 | 必要参数 |
|--------|---------|---------|
| lock | 锁定设备屏幕 | mainText, subText |
| unlock | 解锁屏幕 | - |
| wipe | 恢复出厂设置 | - |
| reboot | 重启设备 | - |
| reset_password | 重置密码 | password |
| install_app | 安装应用 | apkPath |
| uninstall_app | 卸载应用 | bundleName |
| disable_wifi | 禁用 WiFi | - |
| enable_wifi | 启用 WiFi | - |
| disable_mobile_data | 禁用移动数据 | - |
| enable_mobile_data | 启用移动数据 | - |
| set_gps | GPS 开关 | enabled |
| set_bluetooth | 蓝牙开关 | enabled |
| disable_camera | 禁用相机 | - |
| enable_camera | 启用相机 | - |
| get_device_info | 查询设备信息 | - |
| get_app_list | 查询应用列表 | - |
| sync_config | 同步配置 | config |

## 五、技术特性

### 5.1 后台服务保活机制

系统使用 `@ohos.ability.backgroundTaskManager` 申请长时任务，防止应用进程被系统回收。在设备息屏或空闲状态下，后台 MQTT 长连接服务仍可维持运行，确保服务器随时可以下发管控指令。

### 5.2 企业管理员激活

应用首次启动时需要激活 EnterpriseAdminExtensionAbility，将应用设置为设备管理员。激活后，应用获得对企业设备管理功能的访问权限，可执行 lock、wipe 等高级操作。管理员激活和禁用操作均会触发回调通知，应用据此更新设备管控状态。

### 5.3 断线重连容错

MQTT 连接支持自动重连配置，配合 ReconnectManager 的智能延迟策略，在各种网络环境下均能自动恢复连接。当设备网络从离线变为在线时，NetworkMonitor 触发回调通知，MQTT 客户端立即尝试重连，最大限度减少命令积压时间。

## 七、错误处理

### 7.1 错误码定义

| 错误码 | 含义 |
|--------|------|
| 201 | 权限验证失败 |
| 401 | 参数错误 |
| 9200001 | 应用不是设备管理员 |
| 9200002 | 管理员无权限执行该操作 |

### 7.2 异常处理策略

命令执行采用 try-catch 捕获 BusinessError 异常，异常信息通过 CommandResult.failure 方法封装后上报服务器，便于运维人员排查问题。MQTT 通信层的错误统一在 onError 回调中记录日志，并通过连接状态监听器通知相关模块触发重连流程。
